Building the accessibility tree

How do browsers translate HTML markup into an accessibility tree?

(There is actually a specification for this with rules that browsers should follow. This is merely a very high-level - and therefore necessarily incomplete and partly inaccurate - overview!)

  • Most of the common HTML elements have an "implicit role".

    The reverse is not true - many roles do not have a corresponding native HTML element. (More on this later!)

  • The name, and the relevant states/properties, can often be easily obtained from the HTML as well

Some examples:

  • <button>Click me</button> has a role of 
    button
    and a name of 
    "click me"
    . Simple stuff!

  • <label for="checkbox">show details</label><input type="checkbox" checked id="checkbox"/> has

    • A role of 
      checkbox
      (
      input
      elements can have a variety of roles depending on their 
      type
      attribute)
    • A name of 
      "show details"
      (from the text of the associated label element)
    • A 
      checked
      state with the value 
      true
      .
      (This is due to the 
      checked
      attribute that's present. Without that the 
      checked
      state would be false.)

Note that elements like <div>some content</div> and <span>some content</span> have no corresponding role! They don't give rise to a node in the accessibility tree, so while their content is still announced by a screenreader, they have no role and therefore no semantic meaning.
