What ARIA doesn't do

It's important to note that the only thing affected by ARIA is the accessibility tree. And this in turn has no relevance except to assistive technologies like screenreaders.

And, as we all know, there is much more to accessibility than just screenreaders.

So it is not true that "ARIA makes pages accessible" - or at least, not as simple as that!

An example

<div role="button">hey I'm a button</div> will be considered to be a button as far as screenreaders are concerned:
  • it will be announced as a button
  • it will appear in a list of buttons on the page
  • users will be given a way to trigger a click action on it
but...
  • this "button" won't be keyboard-focusable (in the tab order)
  • even if you make it so (with tabindex="0"), that focus won't be automatically visible
  • it won't allow keyboard activation (with space or enter key) like a native button does

You could do all this yourself (with CSS and Javascript) - but a native HTML <button> would give you all of this for free, as well as working the same with screenreaders!

Similarly, something like <span role="checkbox" aria-label="I'm a checkbox"></span> might make an assistive technology treat it as a checkbox - but without considerable extra effort it will neither look like a checkbox to sighted users nor do anything at all when clicked/activated!

These examples are not particularly realistic (breaking the "first rule of ARIA" - see later!), but they make an important point that's relevant also to more appropriate uses of ARIA:

ARIA only deals with semantics - what an element means. It's also static, like all HTML - it can only provide the state of a UI element at any one time. And it doesn't do anything at all for keyboard accessibility.

All that has to be provided separately via Javascript.
