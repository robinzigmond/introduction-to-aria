Content of the accessibility tree

Essentially, the accessibility tree consists of a collection of linked "accessibility objects". That is, each element on the page is interpreted as such an object - that is, an element with certain properties/features.

These features consist of:

  • A role

    Essentially says what the element is and/or what it does. (Slightly more formally, this is the element's semantic meaning.) Examples of roles include:

    • heading
    • button
    • combobox
    • and many more as we shall see!

    Roles are used by assistive technologies in many ways:

    • To announce what you are interacting with - eg "you are on a button"
    • Allowing standard ways to interact with it if possible (eg giving the user a way to "click" if it's a button)
    • To include it in summaries of information on the page - eg a list of landmarks/headings/links etc.

  • A name

    Sometimes called "accessible name" to make its purpose clear, this is what is announced by the screenreader as a summary of what the element is or does.

    Also important for voice recognition software - if the user says "click read more", then it will look for a clickable element with name read more.

  • Various other states and properties of the element.

    • This is a catch-all for any and all of many other things which might need to be known about the element.
    • As a simple example: for a checkbox the user will want to know whether it is currently checked or not. This is an example of a state.
    • There's no clear-cut distinction between states and properties, but essentially a state is something that's expected to change as the user interacts with the page, while a property isn't.
    • The available states and properties depend on the element's role.
